# Prompt / Skill 可复用模板

> 跨需求复用的提示词模板。换行业也能再跑一遍这条"需求 → 产品"流水线。完整方法论见 [ai-delivery-methodology.md](./ai-delivery-methodology.md)。

## ① 需求澄清模板

```text
客户原话："<贴原话>"。在我提问前，先别写代码。请：
1. 把每个名词翻译成"数据动作 + 涉及实体/字段"。
2. 列出隐含决策，每条给"我的默认值 + 一句理由"。
3. 标出哪些必须等客户确认、哪些可先用默认值跑起来。
4. 给一条"会让方案失败"的风险，以及如何先堵住。
不要扩大范围，只对齐这段需求。
```

## ② Skill：按设计规范生成页面

```text
按以下规范生成 <页面名>：
- 角色：<warehouse|buyer|admin>，本页服务的高频决策：<...>
- 信息密度对齐该决策；必须包含 空态 / 载入骨架 / 错误态 三态。
- 复用设计 token（配色 / 间距 / 圆角 / 阴影），不硬编码颜色。
- 数据只经 typed tool / server action，不直接拼 SQL。
- 涉及金额用整数分、展示层 /100；敏感字段按角色脱敏。
输出：组件代码 + 一句"它如何体现'界面有逻辑'"。
```

## ③ Skill：生成服装 SKU mock 数据

```text
生成服装批发 mock：
- 8 个款号，每款 3~4 色 × 3~5 码（S/M/L/XL/2XL），约 200 个 SKU。
- 每个 SKU 一条可回放流水链：期初 + 若干入库/出库，满足 I2（库存=流水累加）。
- 金额用整数分。
- 按 docs/design/stocktake-reconciliation.md 编排 9 个差异 SKU、6 类成因，
  每条在流水/单据里留好对应证据（串色配对、同号重复、收货==应收未质检、快照后流水）。
```

## ④ Skill：按评分卡自检

```text
用 docs/reference/evaluation-rubric.md 的六维评分卡给当前版本打分：
1. 逐维给"分数 + 一句证据"。
2. 只列不达标项 + 最小修复动作（不复述达标项）。
3. 末尾给总分，并标出【阻塞项（必须修）】与【优化项】。
另跑 RC-01..09 与 AI 发布红线，任一不过则结论为"不可发布"。
```
