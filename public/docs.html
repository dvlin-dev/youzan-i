AI-Native 进销存系统 · 服装批发

云链进销存

把库存从「可被覆盖的数字」重构为由不可变流水累加出的派生值——用守恒不变量 + 待复核审批闸，从结构上消灭错账；并让 AI 经类型化工具层成为系统的一等操作者。

“我们做服装批发，仓库 8 个人用 Excel 记库存，上月盘点差了三万多。” —— 客户原话，也是这个项目要解决的核心问题

问题

病根不在“算错”，在 Excel 的数据模型

把库存存成“一个可以随手改的数字”，决定了它一定会漂移。三个结构性缺陷：

缺陷 01

可被覆盖

库存是一个能直接改写的数字，改了就没了原值——没有“为什么从 120 变成 95”。

缺陷 02

无留痕

谁、何时、依据哪张单改的，查无可查；月底对不上时，没人能复盘“差在哪”。

缺陷 03

无复核

8 人并发录入、没有第二双眼，单点错误悄悄进账、层层累积。

结果：账面与实物逐月漂移 → 盘点“差三万”，而且不知道差在哪、也追不回。
解法立意

库存是“派生值”，不是“存储值”

一个根本转变：不再存库存数字，而是只追加不可变流水，库存由流水实时累加得出。

Excel / 传统 CRUD
stock.qty = 120
← UPDATE 可覆盖，原值丢失
→
云链 · append-only ledger
库存 = Σ ledger.delta
← 只能追加流水，永不覆盖
I1　期初 + 入库 − 出库 = 期末 I2　库存 = 流水累加（单一真相） 纠错只能红冲，原始错误永久留痕
错账不再“被覆盖消失”，而是“留痕、可复盘、可追溯”——这就是治“差三万”的物理基础。
产品全景

一条主链路，三个角色

入库采购到货 / 录入→ 库存预警低库存 / 断码→ 采购补货采购单状态机→ 出库销售 / 调拨→ 盘点对账AI 两层归因

库存矩阵

款号 × 颜色 × 尺码三维，点任意格子看该 SKU 的全部流水链。

出入库 + 审批

像 Excel 一样矩阵录入；改库先进“待复核”，审批后才入账。

采购单状态机

草稿 → 已下单 → 部分到货 → 已入库；到货量由流水派生回写。

盘点对账

账面快照 vs 实盘，AI 把差异拆成可执行的几摞。

AI 助手

一句话查库存 / 连续出入库 / 对账，全程经类型化工具。

数据层权限

三角色 RBAC，成本价对仓管字段级脱敏。

仓管 · 只看库存 + 录出入库（成本脱敏） 采购 · 采购单 + 对账 老板 · 全局 + 盘点过账

AI-Native

自然语言 → 连续操作 → 审批入账

AI 在一轮对话里能连续完成多步（agent loop），写操作直接生成待复核单，全程把思考 / 工具调用 / 回答实时流式展示。

一句话“快断货的都补到 30”→ Agent 一轮多步low_stock + 5×record_move→ 生成待复核单不直接落库→ 任意人审批单人即可→ 入账守恒护栏兜底

只调类型化工具

AI 不写裸 SQL、不直接落库；输出只能映射到白名单 tool()。

审批闸 = HITL

写操作先进待复核、人工审批后才入账——既是治错账的复核，也是 AI 的安全闸。

第 2 层归因解释

差异抽屉里 LLM 基于第 1 层证据排序解释，降级安全、不改结论。

“看看哪些快断货了，都补到 30” → 一轮内 5 张入库待复核单：米白/S +30、墨绿/M +18 …
硬约束（不可妥协）

把“正确”焊进结构，而不是靠自觉

✓
库存不可直接编辑，只能追加流水；无 UPDATE stock.qty 入口。
✓
流水只增不改不删；纠错用红冲，原始错误永久留痕。
✓
守恒不变量随时成立：I1 / I2。
✓
改库需审批入账（单人即可）；入账经原子守恒护栏，并发出库也打不穿。
✓
金额一律整数分，仅展示层 /100。
✓
待复核草稿存于独立 move_draft，与不可变流水物理隔离。
✓
AI 不写裸 SQL、不直接落库，只走类型化工具。
✓
权限在数据层生效：RBAC + 成本字段脱敏（不进响应体）。

一个统一论点

为让 AI 安全所必须建的地基，恰好就是治“盘点差三万”本就该建的那一套。

不可变流水当审计 · 库存守恒当护栏 · 待复核审批当 human-in-the-loop · 类型化工具当边界——好的传统架构与 AI-Native 架构，在这里收敛为一个。换个行业（餐饮 / 零售），把“款 × 色 × 码”换成对应维度、重写守恒不变量，这套流程能原样再跑一遍。
