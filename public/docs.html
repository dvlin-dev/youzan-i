AI-Native 进销存系统 · 服装批发

把库存从「可被覆盖的数字」
重构为「不可变流水的派生值」

用一组守恒不变量 + 待复核审批闸，从结构上消灭错账；并让 AI 经类型化工具层成为系统的一等操作者——而它所需的安全地基，恰好就是治“盘点差三万”本就该建的那一套。

“我们做服装批发，仓库 8 个人用 Excel 记库存，上月盘点差了三万多。” —— 客户原话，也是这个项目要解决的核心问题

问题

病根不在“算错”，在 Excel 的数据模型

把库存存成“一个可以随手改的数字”，就决定了它一定会漂移。三个结构性缺陷叠加，月底必然对不上、且追不回。

缺陷 01 · 可被覆盖

改了就没了原值

库存是能直接改写的数字——没有“为什么从 120 变成 95”，旧值与依据一并丢失。

缺陷 02 · 无留痕

查不到谁改的

谁、何时、依据哪张单改的，无从追溯；对不上时没人能复盘“差在哪”。

缺陷 03 · 无复核

错误悄悄进账

8 人并发录入、没有第二双眼，单点错误层层累积成系统性偏差。

结果 = 账面与实物逐月漂移 → 盘点“差三万”，而且不知道差在哪、也追不回。
解法立意

库存是“派生值”，不是“存储值”

一个根本转变：不再存库存数字，而是只追加不可变流水，库存由流水实时累加得出。错账不再被覆盖消失，而是留痕、可复盘、可追溯。

Excel / 传统 CRUD
stock.qty = 120 ← UPDATE 可覆盖，原值丢失
→
云链 · append-only ledger
库存 = Σ ledger.delta ← 只能追加流水，永不覆盖
I1　期初 + 入库 − 出库 = 期末 I2　库存 = 流水累加（单一真相） 纠错只能红冲，原始错误永久留痕

把“正确”焊进结构，而不是靠自觉——硬约束

✓
库存不可直接编辑，只能追加流水；无 UPDATE stock.qty 入口。
✓
流水只增不改不删，经手人外键到真实用户；纠错用红冲，原始错误永久留痕、可追溯到谁。
✓
改库需审批入账（单人即可）；入账经原子守恒护栏，并发出库也打不穿。
✓
待复核草稿存于独立 move_draft，与不可变流水物理隔离。
✓
金额一律整数分，仅展示层 /100。
✓
权限在数据层：RBAC + 成本字段脱敏（不进响应体）。

采购 / 老板按 款 × 色 × 码 批量建 SKU、改安全库存阈值；新款库存从 0 起（建档不造流水）。

库存矩阵

款号 × 颜色 × 尺码三维；点任意格子看该 SKU 的全部流水链。

出入库 + 审批

像 Excel 一样矩阵录入；改库先进“待复核”，审批后才入账。

采购单状态机

新建采购单 → 下单 → 登记到货；草稿→已下单→部分到货→已入库，到货量由流水派生回写。

盘点对账

发起盘点（账面快照取当前库存）→ 录实盘 → AI 把差异拆成可执行的几摞。

AI 助手

一句话查库存 / 连续出入库 / 对账，全程经类型化工具。

仓管 · 只看库存 + 录出入库（成本脱敏） 采购 · 商品档案 + 采购单 + 对账 老板 · 全局 + 盘点过账

AI-Native

自然语言 → 连续操作 → 审批入账

AI 在一轮对话里能连续完成多步（agent loop），写操作直接生成待复核单，全程把思考 / 工具调用 / 回答实时流式展示。

一句话“快断货的都补到 30”→ Agent 一轮多步low_stock + 5×record_move→ 生成待复核单不直接落库→ 任意人审批单人即可→ 入账守恒护栏兜底

只调类型化工具

AI 不写裸 SQL、不直接落库；输出只能映射到白名单 tool()。

审批闸 = HITL

写操作先进待复核、人工审批后才入账——既是治错账的复核，也是 AI 的安全闸。

第 2 层归因解释

差异抽屉里 LLM 基于第 1 层证据排序解释，降级安全、不改结论。

“看看哪些快断货了，都补到 30” → 一轮内 5 张入库待复核单：米白/S +30、墨绿/M +18 …（实时显示 思考 → 查询低库存 ✓ → 逐笔登记 ✓ → 流式回答）

Agent 的工具集（底层能力）

工具读/写参数作用谁能用
query_stock读styleNo, color, size查某 SKU 当前库存（= posted 流水累加）。全部
low_stock读—列出低于安全库存（含断码）的 SKU，带 款 / 色 / 码 / 当前 / 安全。全部
recon_summary读—盘点对账归因汇总：盘亏毛额 / 真损失 / 可追回 / 各成因分桶。采购 · 老板
record_move写type, styleNo, color, size, qty登记一笔入 / 出库 → 生成待复核单（不直接落库）；一轮可多次。仓管 · 老板
query_sql读sql（单条 SELECT）跑受控只读 SQL 查任意数据——兜住长尾问题，不被预置工具限制；独立只读角色物理拒写 + 按角色脱敏。按角色

这些是 lib/ai/tools.ts 暴露给 AI 的白名单工具：Zod 入参（→ JSON schema 喂给模型）+ 服务端 execute（守恒 / 权限 / 审计在内部）。它们按角色挂载、是类型化工具层 lib/actions.ts 的子集，写操作只生成待复核单。盘点第 2 层归因解释 explainDiff 由差异抽屉按需触发，不在对话工具集内。

只读 SQL 工具（query_sql）：给 AI 一个“读任意数据”的兜底能力——它能跑一条 SELECT 回答预置工具覆盖不到的长尾问题，但只读不写，靠纵深防御三层：① 语句层只允许单条 SELECT（拒 INSERT/UPDATE/DELETE/DDL/多语句/注释/危险函数）；② 连接层用独立只读角色（仅 GRANT SELECT），即使解析被绕过 DB 也物理拒写——仓管连接的 sku 还是去掉 cost_price 的视图，连 to_jsonb(sku) 也带不出成本价；③ 字段脱敏焊在 DB 层 + 语句超时 + LIMIT + 每次调用落审计表 query_audit（被审计者读不到自己的审计）。要显示经手人姓名时经脱敏视图 app_user_public（只有 id / 名 / 角色，无邮箱口令）join，口令表本体只读角色 DB 层拒读。既保住灵活性，又把“只读 + 不泄露受限数据”焊在数据层（设计见 design/ai-native-architecture）。

交付方法论

从一句模糊需求，到可上线的产品

不止“用 AI 写代码”，而是一套可复现的“需求 → 产品”交付流程：原型先行、文档对齐、知识库治理、可回归评估。下面拆成五件事——拆解编排、工具链路、翻车救场、工作流迭代、评估判断。

Q1

拆解与编排

怎么把模糊描述拆成 AI 能执行的输入？

先给一个大概的需求 → 让 AI 联网调研（社区实践 + 论文，给方案找先验）→ 让它先出一个高保真原型 → 基于原型快速对齐成想要的样子 → 再把稳定结论沉淀成架构与知识库。关键：不在“写完一大坨代码”之后才对齐，而是用低成本的原型先把方向与数据模型敲定。

把模糊原话翻译成“数据动作 + 默认值 + 阻塞分类”，靠的是一套有治理规则的知识库：根 CLAUDE.md 路由事实源，docs/{design, plan, reference} 分层维护——新需求先写 plan/、稳定结论回写 design/ 或 reference/、同一主题只保留一个事实源。于是每一次拆解都基于同一份稳定上下文，而不是临时拍脑袋（详见「研发知识库」一节）。

Q3

翻车与救场

AI 哪里翻车了，怎么救的？

核心心法：让错误在便宜的阶段暴露——原型 > plan > 代码。三道防线：

防线 1 · 原型期

抓方向 / 理解偏差

趁原型成本低，先把“实盘到底进不进状态”“对账差异从哪来”这类大方向敲定（结论：实盘是一次性校准输入、不是状态）。

防线 2 · plan 期

定稿后人工 review

plan 文档定好后人工复核，在写大量代码之前把理解对齐、把偏差讨论清楚。

防线 3 · 代码后

文档先行再改码

仍有漏聊 / 没做对的，基于当前版修改——且先回写文档、再改代码。

真实例子：外部 agent 评审发现 并发出库可打穿库存 / 采购状态机账实不一致 / 成本脱敏不在数据层 / append-only 名不副实，逐条修复（原子守恒护栏 postDraftAtomic、草稿表物理隔离 move_draft、maskCost）；以及 @openai/agents 不暴露网关 reasoning → 改手写流式循环。

Q4

工作流迭代

再做一遍会怎么改工作流？

这套工作流本身就是迭代沉淀出来的，下次直接复用；换行业只换领域维度（款 × 色 × 码 → 对应维度）与守恒不变量，流程原样再跑：

  1. 模糊需求 → AI 联网调研（社区实践 + 论文）
  2. 高保真原型 → 基于原型对齐方向（低成本纠偏，敲定数据模型）
  3. plan 文档 → 人工 review（写大量代码前对齐理解）
  4. 知识库沉淀（CLAUDE.md + docs/{design, plan, reference}，单一事实源）
  5. 生产实现（类型化工具 + 守恒 + 测试 + 一键部署）
  6. 迭代回路：发现问题 → 基于当前版改 → 先回写文档、再改代码 → 测试 / 部署 → 稳定事实回 design / reference
Q5

评估与判断

怎么判断这版比上一版好？靠感觉还是可重复标准？

原型期偏人工判断，但脑子里有明确的执行 case——主链路（入库 → 预警 → 采购 → 盘点）必须能跑通。落地期转为三层可重复评估（前两层是工具链路，第三层是页面级）：

① 工具链路 · 单工具测试

每个 tool 逻辑正确

固定数据的测试用例，验证工具本身——Vitest：库存守恒 I1 / I2 + 6 类归因检测器回归。

② 工具链路 · AI 端到端

AI 能否正确编排工具

Mock 一批场景，让 AI 驱动整条工具流程，断言它调对了工具——与跑测试用例同理，只是改由 AI 跑。

③ 页面级 · AI 测页面

看页面真实效果

用 AI（agent-browser）操作页面、核对效果——预填登录 / RBAC 脱敏 / 盘点数 / copilot 全链路。

再加六维评分卡 + RC-01..09 回归用例作为“好不好”的可重复标准（见 reference/evaluation-rubric）。一句话：好不好是算出来的，不是感觉出来的。

一个统一论点

为让 AI 安全所必须建的地基，恰好就是治“盘点差三万”本就该建的那一套。

不可变流水当审计 · 库存守恒当护栏 · 待复核审批当 human-in-the-loop · 类型化工具当边界——好的传统架构与 AI-Native 架构，在这里收敛为一个。换个行业（餐饮 / 零售），把“款 × 色 × 码”换成对应维度、重写守恒不变量，这套流程能原样再跑一遍。
